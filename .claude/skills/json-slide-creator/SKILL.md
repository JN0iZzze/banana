# JSON Slide Creator

Скилл для работы только с новой архитектурой слайдов на `JsonSlideRenderer`.

Подробный контракт и примеры находятся в `src/presentation/json-renderer/README.md`. Сначала следуй этому скиллу как краткому рабочему протоколу, а за деталями и точными примерами формы JSON обращайся в `src/presentation/json-renderer/README.md`.

## Когда применять

Применяй этот скилл при любых запросах:

- создать новый слайд
- изменить существующий слайд
- найти слайд по номеру, названию, `id` или пути к схеме
- обновить структуру, тексты, карточки, layout или JSON-схему слайда
- усилить заголовок, `lead`, подписи, формулировки и главный тезис
- выстроить более ясный сторителлинг и порядок аргументов внутри слайда

## Главное правило

Работай через новую JSON-архитектуру, только если целевой слайд использует `JsonSlideRenderer`.

Если запрос относится к legacy-слайду, который рендерится обычным `.tsx` компонентом, не вноси правки сразу. Сначала:

1. Явно сообщи, что это legacy-слайд, а не JSON-renderer слайд.
2. Спроси, что именно хочет пользователь:
   - точечно править legacy-слайд как есть
   - перевести его на `JsonSlideRenderer`
3. До начала правок дай конкретные замечания и предложения:
   - можно ли выразить слайд через существующий JSON-контракт
   - какой layout подходит
   - что не помещается в текущие ограничения JSON
   - есть ли риск, что миграция будет искусственной и лучше оставить `.tsx`

Без такого подтверждения legacy-файл не редактируй.

## Как определить целевой слайд

Пользователь может ссылаться на слайд разными способами. Разрешай цель в таком порядке:

1. Если дан явный путь к JSON-файлу, работай с ним напрямую.
2. Если дан путь к `.tsx` файлу слайда, проверь, это JSON-слайд или legacy.
3. Если пользователь указывает номер, название или `id`, найди запись в `src/presentation/decks/mainDeck.ts`.
4. Если в `mainDeck.ts` у записи `component: JsonSlideRenderer`, найди связанный документ в `src/presentation/jsonSlideDocumentRegistry.ts`.
5. Если соответствие неоднозначно, сначала уточни у пользователя, а не гадай.

## Рабочий процесс для JSON-слайда

### Новый слайд

1. Создай JSON-схему в `src/presentation/schemas/`. Для `backdrop.variant` **`spotlight`** или **`none`** добавь `"borderFrame": true`, если пользователь явно не просит слайд без декоративной рамки. Для **`grid`** и **`mesh`** рамка уже внутри `SlideBackdrop` — **не** ставь `borderFrame: true` (иначе смысловой дубль; рантайм в `JsonSlideShell` всё равно не рисует вторую рамку).
2. Зарегистрируй документ в `src/presentation/jsonSlideDocumentRegistry.ts`.
3. Добавь запись в `src/presentation/decks/mainDeck.ts` с `component: JsonSlideRenderer`.
4. Заполняй только поддерживаемые поля контракта.
5. Проверь `npm run typecheck` и `npm run build`.

### Редактирование существующего JSON-слайда

1. Найди запись в `mainDeck.ts`.
2. Найди связанную JSON-схему через `jsonSlideDocumentRegistry.ts` или прямой путь пользователя.
3. Вноси изменения прежде всего в JSON-схему.
4. Меняй `mainDeck.ts` только если действительно нужно обновить `id`, `title`, `notes`, `hidden` или саму регистрацию.
5. Если для задачи не хватает текущего контракта, остановись и сначала объясни ограничение пользователю.

## Краткий контракт JSON Renderer

Полный контракт см. в `src/presentation/json-renderer/README.md`. Ниже только базовые правила, которые нужно помнить всегда.

### Верхний уровень документа

Допустимые top-level поля:

- `frame`
- `backdrop`
- `content`
- `header`
- `layout`

Не придумывай дополнительные секции.

### Поддерживаемые top-level опции

- `frame.align`: `top` | `center` | `bottom`
- `frame.padding`: `compact` | `default` | `spacious`
- `backdrop.variant`: `grid` | `mesh` | `spotlight` | `none`
- **`backdrop.borderFrame`:** для **`spotlight`** / **`none`** ставь `"borderFrame": true`, если пользователь явно не просит слайд без рамки. Для **`grid`** / **`mesh`** не ставь — рамка уже в `SlideBackdrop` (см. `layout.tsx` и `JsonSlideShell`).
- `content.width`: `full` | `wide` | `content` | `narrow`
- `content.density`: `compact` | `comfortable` | `relaxed`
- `content.align`: `left` | `center`
- `header.meta`: обязательная строка
- `header.title`: опциональная строка (можно оставить только `meta` для слайдов без крупного заголовка)
- `header.lead`: опциональная строка

### Layout presets

Разрешены только:

- `asymmetricColumns`
- `equalColumns`
- `bentoGrid`
- `uniformGrid`
- `splitLayout` (регионы: `card`, `text`, `layout` или `quote`; справа/слева часто вкладывают `mediaGallery`)
- `stackLayout` (вертикальный стек тех же регионов; `span` в сумме 12)
- `mediaGallery` (корень или внутри split/stack)

Правила выбора:

- `asymmetricColumns` для явной разницы ширин, например `7+5` или `8+4`
- `equalColumns` для равных колонок, например `6+6` или `4+4+4`
- `bentoGrid` для ручного позиционирования ячеек с `colStart`, `rowStart`, `colSpan`, `rowSpan`
- `uniformGrid` для повторяемой сетки одинаковых карточек без ручных координат

Не изобретай новые layout type без расширения рантайма и документации.

## Карточки

Старые поля `overline`, `title`, `body` на уровне card считаются устаревшими. Используй новый card contract.

Основные поля карточки:

- `tone`: `standard` | `accent`
- `padding`: `compact` | `default` | `spacious`
- `stackGap`: `xs` | `sm` | `md` | `lg`
- `leadingIcon`: только id из registry
- `watermarkIcon`: только id из registry
- `subtitle`: короткая верхняя подпись
- `justify`: `start` | `end` | `between`
- `items`: обязательный непустой массив

### Что помнить про `subtitle`

- `subtitle` живёт отдельно над основным стеком
- `subtitle` не участвует в `justify`
- для pinned label предпочитай `subtitle`
- если подпись должна быть частью общего стека, используй обычный text item в `items[]`

### `items[]`

Элементы бывают двух типов:

1. Текстовый item:
   - `{ "variant": "...", "text": "..." }`
2. Компонентный item:
   - `{ "type": "component", "component": "<id>", ... }`

Поддерживаемые текстовые `variant`:

- `overline`
- `caption`
- `h2`
- `h3`
- `body`
- `bodyLg`

Не используй в card items:

- `h1`
- `lead`
- `meta`
- `tileAccent`

Порядок контента задаётся только порядком массива `items[]`. Не предполагай, что заголовок всегда первый, а body всегда последний.

## Поддерживаемые component items

Сейчас component rows в карточках не произвольные. Можно использовать только allowlist из текущего рантайма.

Базово ориентируйся на:

- `tagList`

Если запрос требует новый component item:

1. Обнови типы в `src/presentation/jsonSlideTypes.ts`.
2. Добавь валидацию в `src/presentation/parseJsonSlideDocument.ts`.
3. Добавь renderer в `src/presentation/json-renderer/jsonSlideCardComponentRegistry.tsx`: компонент, запись в `JSON_SLIDE_CARD_COMPONENT_REGISTRY` и ветку в `renderJsonCardComponentItem()` (см. README). `JsonCardNode.tsx` для этого не расширяй.
4. Задокументируй форму в `src/presentation/json-renderer/README.md`.

Не добавляй свободные `Record<string, unknown>` пропсы и не обходи parser.

## Иконки карточек

`leadingIcon` и `watermarkIcon` берутся только из registry и allowlist.

Если нужной иконки нет:

1. Не подставляй произвольную строку.
2. Сначала расширь registry и типы.
3. Затем используй новый id в JSON.

## Правила текста и сторителлинга

Эти правила применяются и при создании нового контента, и при ревью существующего. Если пользователь даёт тексты для слайда, не только раскладывай их по JSON, но и редактируй под ясный тезис, удобное чтение со сцены и последовательную подачу.

### Заголовок — утверждение, а не тема

`header.title` должен сообщать главный вывод слайда, а не просто называть тему.

- Плохо: `"Анализ рынка"`, `"Результаты Q3"`, `"Технологический стек"`
- Хорошо: `"Рынок вырос на 40% за счёт B2B-сегмента"`, `"Q3: выручка превысила план на 18%"`, `"Монолит тормозит масштабирование — нужна миграция на сервисы"`

Если аудитория прочитает только заголовки всех слайдов — она должна понять суть всей презентации.

`header.lead` используй для одного сильного поддерживающего тезиса: он может раскрывать вывод, давать контекст, объяснять причину или подводить к следующему блоку.

### Один слайд — одна мысль

Каждый слайд доказывает ровно один тезис из заголовка. Если хочется добавить "и ещё кое-что" — это второй слайд.

### Сторителлинг важнее механической краткости

Презентация — не UI-микрокопирайт. Здесь допустимы полноценные предложения, короткие абзацы и объяснения в 3-4 строки, если они реально помогают понять мысль, причинно-следственную связь или переход между тезисами.

- `header.title`, `subtitle`, `overline`, `caption` и короткие карточки держи компактными.
- `header.lead`, `body`, `bodyLg`, `quote` и текстовые регионы могут быть полными предложениями.
- Лучше один сильный объясняющий блок на 2-4 строки, чем набор обрубков, из которых неясно, в чём посыл.
- Если текст уже превращается в стену или уносит в сторону вторую мысль, разбей его на два блока или два слайда.

### Плотность текста

Аудитория не должна разрываться между чтением и речью спикера. Текст на слайде помогает схватить мысль и запомнить аргументы, а не заменяет весь рассказ.

- Обычно держи слайд в пределах **до 6 карточек или буллетов**.
- Короткие карточки и буллеты по умолчанию делай короткими, но не жертвуй смыслом ради жёсткого лимита слов.
- Для поясняющего текста допустимы **1-2 полных предложения** или компактный абзац в **3-4 строки**, если без этого рушится логика.
- Если текста стало слишком много, разбей слайд на два, собери аргументы в более жёсткую структуру или замени часть текста визуалом.

### Суть вперёд, без воды

Бери из инфостиля только то, что полезно для презентации:

- начинай с ответа, а не с разгона;
- убирай канцелярит, общие слова и пустые усилители;
- не прячь важную мысль в конец абзаца или карточки;
- предпочитай факты, числа, контраст и причинно-следственные связи вместо туманных формулировок;
- если термин, англицизм или техническая деталь не помогают тезису, упрощай формулировку.

### Порядок подачи

Веди сверху вниз: сначала ответ (`header.title`), потом аргументы (карточки), потом детали внутри карточек. Не прячь вывод в конец слайда.

### Тон и формулировки

- Пиши по-человечески, но без разговорной расхлябанности.
- Не надувай слайд маркетинговыми эпитетами и искусственной важностью.
- Каноничные названия продуктов, моделей, компаний и технологий не переводи произвольно.
- В технических презентациях допустимы термины и полные предложения, если они действительно продвигают мысль, а не демонстрируют "умность" текста.

## Практические правила авторинга

- Предпочитай короткий и строгий JSON-контракт вместо произвольной вложенности.
- Для высоких карточек с заголовком и описанием часто подходит `justify: "between"`.
- Для компактных карточек чаще нужен `h3`, для крупных блоков чаще `h2`.
- `body` и `bodyLg` можно использовать для полных предложений, если нужен ясный аргумент, а не телеграфный обрывок.
- `bodyLg` используй там, где контента мало, но supporting text должен читаться как заметная мысль.
- Если смысл держится только на голосе спикера, а со слайда его не понять, усили заголовок, `lead` или один из текстовых блоков.
- Используй только поддерживаемые поля из контракта, даже если очень хочется добавить `className`, inline-стили или произвольный React.

## Жёсткие ограничения

- Не добавляй responsive Tailwind-префиксы вроде `sm:`, `md:`, `lg:` в этом пути.
- Не добавляй arbitrary `className` support в JSON.
- Не добавляй markdown, media, arbitrary React nodes или свободные nested blocks в `card.items`.
- Не расширяй контракт молча: если нужно новое поведение, явно обновляй типы, parser, renderer и документацию.

Это особенно важно, потому что презентация работает как fixed-stage, а не как responsive page.

## Когда не надо тащить задачу в JSON

Не пытайся насильно уложить слайд в `JsonSlideRenderer`, если ему нужны:

- изображения или видео со специальной композицией
- графики, виджеты, кастомные React-компоненты
- условные ветки рендера
- layout, который не выражается через текущие presets
- иконко- или media-специфичная логика

В таком случае честно скажи, что лучше оставить или сделать отдельный `.tsx` слайд.

## Обязательное поведение при legacy-запросах

Если пользователь просит изменить существующий legacy-слайд:

1. Не начинай с правок файла.
2. Сначала кратко разберись, что в нём есть:
   - какой layout
   - есть ли изображения, видео, кастомная композиция
   - возможно ли выразить это через текущий JSON-контракт
3. Дай пользователю конкретные предложения:
   - оставить legacy и внести локальные правки
   - мигрировать на JSON без потери смысла
   - частично упростить структуру ради миграции
4. Только после явного ответа пользователя переходи к изменениям.

## Чеклист перед завершением

**Контент:**
- `header.title` — это утверждение/вывод, а не просто название темы
- Один слайд — одна главная мысль и понятный шаг в общей истории
- На слайде нет визуального и смыслового перегруза; длинный текст оправдан только там, где он реально объясняет тезис
- Короткие карточки и буллеты остаются компактными, а длинные `lead`/`body`/`bodyLg` не расползаются в стену текста

**Техника:**
- Для `spotlight`/`none` в схеме есть `backdrop.borderFrame: true`, если не договорились о слайде без рамки; для `grid`/`mesh` поле `borderFrame` не нужно
- Найден правильный слайд по `mainDeck.ts`, `id`, названию, номеру или пути
- Проверено, JSON это или legacy
- Для JSON-слайда использованы только поддерживаемые поля
- При необходимости обновлены registry и parser, а не только схема
- `mainDeck.ts` и `jsonSlideDocumentRegistry.ts` синхронны
- Нет responsive-префиксов и произвольных классов
- Прогнаны `npm run typecheck` и `npm run build`

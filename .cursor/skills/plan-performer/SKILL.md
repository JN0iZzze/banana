# Plan performer

План → оркестратор → этапы субагентами → (при правках JSON-схем слайдов: проход tone-of-voice отдельным Task) → финальная проверка (`npm run typecheck` + `npm run build`).

## Назначение

Скилл задаёт **жёсткий порядок работы родительского агента** (основной чат): сначала получить от **Plan Orchestrator** декомпозицию и готовые промпты для Task, затем **ни в коем случае не смешивать этапы** — вызывать **отдельный** Task на каждый этап; при правках JSON-схем слайдов `src/presentation/decks/<deckId>/schemas/**/*.json` — **отдельный** Task с требованием следовать скиллу **json-slide-tone-of-voice** (шаг 2.5); в конце — **отдельная** финальная проверка (`npm run typecheck` и `npm run build`; при UI-правках — поднять `npm run dev` и прокликать).

Спецификация оркестратора: [.cursor/agents/plan-orchestrator.md](../../agents/plan-orchestrator.md).

## Когда применять

- Пользователь дал **план** (файл, вставку, md-реестр, список задач) на крупную колоду или много файлов.
- Явные формулировки: «plan performer», «через оркестратор», «субагент на этап», «не одним промптом», «после плана — делегирование».

## Workflow (обязательно по шагам)

### Шаг 1 — Оркестрация (один вызов Task)

1. Собери вход для оркестратора: путь к файлу плана/реестра и/или полный текст, цель, ограничения из чата (фокус на деке `/vibecoding`, md-режим, что уже сделано).
2. Вызови **Task** с:
   - `subagent_type`: **`plan-orchestrator`**
   - `prompt`: цель, вложенный/процитированный план, путь к репозиторию `/Users/jn0izzze/OtherAIPro/newgen`, что уже сделано в ветке.
3. Дождись ответа. Если оркестратор пишет, что **не хватает данных** — задай пользователю **1–3** уточнения или дополни контекст и **повтори шаг 1**, пока этапы и промпты не станут исполнимыми.

**Запрещено** на шаге 1 самому «угадывать» декомпозицию вместо оркестратора, если пользователь ожидает этот скилл.

### Шаг 2 — Исполнение по этапам (строго один Task = один этап)

1. Возьми из ответа оркестратора **волны** и для **каждого этапа** — блок с `subagent_type`, `readonly` (если есть) и текст **`prompt` между разделителями `---`**.
2. Для **каждого этапа** отдельно вызови **Task**:
   - подставь указанный **`subagent_type`** (или осмысленную замену только если оркестратор явно ошибся — тогда кратко зафиксируй причину в чате);
   - **`prompt` копируй целиком** из выдачи оркестратора (субагент не видит историю чата);
   - **`model`**: как указал оркестратор; если не указан — по умолчанию уместен быстрый вариант для простых этапов.
3. **Параллельность:** этапы из **одной волны**, которые оркестратор пометил как параллельные, можно запускать **несколькими Task в одном сообщении** (параллельные вызовы). Этапы из разных волн — **только последовательно**: следующая волна после того, как предыдущая закрыта по результатам.
4. **Запрещено:** один Task с текстом «сделай этапы 2–5», «объедини этапы», «продолжи со второго» без нового самодостаточного промпта. Каждый этап — **отдельный** вызов с **своим** полным промптом из плана оркестратора.

После каждой волны кратко зафиксируй в чате: какие слайды/файлы закрыты по факту, открытые риски, блокеры для следующей волны.

### Шаг 2.5 — Tone-of-voice проход по JSON-схемам слайдов (отдельный Task, обязательно при правках)

Если в ходе плана менялись **JSON-схемы слайдов** под **`src/presentation/decks/**/schemas/**/*.json`** (или собирался `jsonDocument` через `defineJsonSlide`), цикл **не считается закрытым**, пока не выполнен этот шаг.

1. Вызови **отдельный** Task — **не** смешивай с этапами оркестратора из шага 2 и **не** объединяй со шагом 3:
   - `subagent_type`: **`general-purpose`** (или **`json-slide-batch-builder`**, если правки затрагивают несколько слайдов одного батча).
   - Промпт субагента обязан требовать: **прочитать и следовать** скиллу **[json-slide-tone-of-voice](../json-slide-tone-of-voice/SKILL.md)**, а также сверяться с `CLAUDE.md` и [src/presentation/json-renderer/README.md](../../../src/presentation/json-renderer/README.md); перечисли изменённые или затронутые JSON-файлы (или дай контекст `git diff`); задача — **проверка и правка пользовательских текстов** (`header.title`, `lead`, карточки, подписи, narrative flow), без выдумывания новых фактов и без правок технической структуры схемы (layout, keys, id иконок), если это явно не запрошено.
2. Порядок: **после** завершения **всех** волн шага 2, **до** шага 3. Если JSON правился в ранней волне, шаг 2.5 всё равно **один раз** после полного прохода этапов.
3. Если субагент внёс правки — при необходимости снова короткий шаг 2.5 или зафиксируй итог; при спорных формулировках — правила в `CLAUDE.md` и прошлые memory-пометки (директивный тон, стрелочные схемы, caption-вариант и т.д.).

### Шаг 3 — Финальная проверка (отдельный Task)

Когда **все** этапы из оркестрации выполнены (или явно отменены/срезаны по согласованию с пользователем) **и** при необходимости отработал **шаг 2.5**:

1. Вызови **ещё один** Task — **только на проверку**, без новых фич:
   - `subagent_type`: чаще всего **`general-purpose`** (обзор диффа, критерии готовности, согласованность с планом/реестром); для прогона команд — допустимо явно попросить субагента выполнить `npm run typecheck` и `npm run build`.
2. В промпте финальной проверки явно перечисли:
   - исходные **критерии готовности** из плана и из этапов;
   - **команды** проверки: **обязательно `npm run typecheck` и `npm run build`** с **exit code 0**; при UI-правках — кратко поднять `npm run dev` и прокликать затронутые слайды;
   - что **не** трогать (scope freeze): только валидация и список находок.
3. Итог для пользователя: краткое резюме + блокеры P0/P1, если есть.

4. **Typecheck/Build (обязательно):** финальный цикл **не считается закрытым**, пока не выполнены **`npm run typecheck`** и **`npm run build`** с **exit code 0** (требование README JSON-рендерера после любой правки JSON-схемы). Родительский агент либо включает это в промпт финального Task, либо выполняет сам после субагента — но **не пропускает**. Результат (команда, exit code) кратко зафиксировать пользователю. При ошибках — исправить, затем снова typecheck/build и при необходимости **повтори шаг 3**.

5. **UI-прогон (если менялись слайды или рендерер):** при изменениях в `src/presentation/decks/**` или `src/presentation/json-renderer/**` убедись, что затронутые слайды корректно рендерятся (`npm run dev`, визуальный прогон, регресс на соседних слайдах в той же деке). Если визуальный прогон невозможен — явно сказать об этом пользователю, не заявлять «готово» вслепую.

Если финальный субагент нашёл **P0** (включая падение typecheck/build) — сначала исправления (снова по этапам или точечным Task), затем **повтори шаг 3** и typecheck/build при необходимости.

## Мини-чеклист качества

- [ ] Был вызван `plan-orchestrator` до начала реализации.
- [ ] На каждый этап — ровно один Task с промптом из оркестрации.
- [ ] Волны соблюдены (нет старта зависимого этапа до готовности зависимостей).
- [ ] После всех этапов — отдельный Task на финальную проверку.
- [ ] Были правки JSON-схем слайдов → выполнен **шаг 2.5** (отдельный Task по скиллу **[json-slide-tone-of-voice](../json-slide-tone-of-voice/SKILL.md)**) или явно зафиксировано, что схемы не менялись.
- [ ] **`npm run typecheck`** и **`npm run build`** завершились с **0** (или явно зафиксировано, что сборка не затронута — редкий случай).
- [ ] При правках UI-слоя — сделан визуальный прогон в `npm run dev` (или явно сказано, что визуальная проверка невозможна).

- Не запускать `npm run build` без причины в режиме планирования — только после реальных правок в коде.

# `/commit` — коммит и push

## Назначение

Зафиксировать осмысленный набор изменений в одном коммите и отправить его в удалённый репозиторий. Не делать пустых коммитов и не маскировать ошибки.

## Порядок действий

1. **`git status -sb`** — убедиться, что есть что коммитить. Если рабочее дерево чистое, остановиться и сообщить: коммит не нужен.
2. **Просмотреть дифф** при необходимости (`git diff`, `git diff --staged`), чтобы сообщение коммита соответствовало фактам.
3. **Индексация** — `git add` только тех путей, которые относятся к текущей задаче. Не добавлять в коммит посторонние файлы (случайные логи, локальные артефакты, секреты). Если пользователь явно просит «всё», можно `git add -A` после проверки `git status`.
4. **Сообщение коммита** — краткое, в нормальном языке: что сделано и зачем (полные предложения, без штампов). При принятой в репозитории конвенции (например Conventional Commits) ей следовать.
5. **`git commit`** с выбранным сообщением. Если после `add` нечего коммитить — не выдумывать пустой коммит; сообщить об этом.
6. **`git push`** — для текущей ветки: обычно `git push` или при первой отправке ветки `git push -u origin <branch>`. Не использовать `--force` без прямого запроса пользователя.

## Ошибки и границы

- Любой ненулевой код выхода у `commit` или `push` — показать вывод команды и не утверждать об успехе.
- Расхождение с remote (нужен pull/rebase) — описать ситуацию; не ломать историю принудительным push.
- Не коммитить то, что явно не должно попадать в git (проверить, что чувствительные файлы не в индексе).
